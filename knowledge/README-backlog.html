Backlog.md — Datei-Modell, Verbindungen & Knowledge-Vergleich

Vollständige Landkarte aller Datei-/Entity-Typen, ihrer Verbindungen (Dependencies, Subtasks, Milestones, References, Documentation), und wie Decisions/Documents entstehen. Plus: Eignung als eine Quelle für Task- und Knowledge-Management pro Repo — verglichen mit dem aktuellen knowledge/-Ansatz.  Quelle: opensrc MrLesk/Backlog.md (main, Stand 2026-06-04).

TL;DR. Backlog verwaltet vier Entity-Typen (task, draft, document, decision) plus milestone als reine Markdown-Dateien unter backlog/. Tasks & Documents haben volle CRUD-Tooling (CLI + MCP + Web). Decisions sind ADRs mit Tooling-Lücke: CLI legt nur ein leeres Skelett an, Inhalt füllst du im File oder Web-Browser. „Eine Quelle pro Repo" ist machbar — weil alles plain-Markdown im Git ist (grep-/qmd-bar, kein Lock-in). Aber: deine lektion-*.md (Lessons-Learned, evidenzlastig) und interaktive *.html (Mermaid) passen nicht in das ADR-Genre decision — die gehören in doc bzw. bleiben Files. Empfehlung unten: Hybrid.

1. Datei- & Ordner-Modell

Alles liegt unter backlog/ im Repo-Root. Jede Entity = eine Markdown-Datei mit YAML-Frontmatter + Body. Dateiname: <prefix>-<id> - <Titel-mit-Bindestrichen>.md.

backlog/
├── config.yml              # Projekt-Config (statuses, labels, DoD-Defaults, prefixes, mcp …)
├── tasks/                  # task-<id> - <titel>.md         (aktive Tasks)
├── drafts/                 # draft-<id> - <titel>.md        (Ideen, nicht im Board)
├── completed/              # task-<id> - …                  (auto-verschoben bei Done, optional)
├── docs/                   # doc-<id> - <titel>.md          (readme|guide|specification|other)
├── decisions/              # decision-<id> - <titel>.md     (ADRs: proposed→accepted→…)
├── milestones/             # m-<id> - <slug>.md             (Single Source of Truth)
├── archive/
│   ├── tasks/   ├── drafts/   └── milestones/             (archiviert, aus Board raus)
└── assets/images/          # Bilder für ![](assets/…) in Tasks/Docs
OrdnerEntityID-SchemaZweck

drafts/Draftdraft-7Roh-Ideen; via promote → Task

Die ID-Prefixes (task) sind in config.yml > prefixes anpassbar (z.B. CCS-). Zero-Padding optional. completed/ wird genutzt, wenn Tasks bei Done dorthin verschoben werden statt im tasks/-Ordner zu bleiben.

4. Wie kommt jede Entity „rein"? (CRUD-Matrix)

Das ist der Kern deiner Frage. Die Tooling-Abdeckung ist stark asymmetrisch.

EntityCLIMCPWeb-BrowserDatei direkt
Taskfull CRUD
create/edit/view/list/archive/demote		fullfullread-only (nie editieren)
Draftcreate/list/view/archive/promoteteilweisejaread-only
Documentcreate/update/list/viewlist/view/create/update/searchfullvia API bevorzugt

Milestonenur list / archivelist/add/rename/remove/archivefull (add/edit/rename/del)Datei = SoT

→ Antwort: Wie kommen Decisions rein?

# 1. Skelett anlegen (CLI) — füllt NUR id/title/date/status
backlog decision create "Submodule auflösen statt Git-Submodul"
#   → erzeugt decisions/decision-N - … .md mit status: proposed
#     und LEEREN Feldern context/decision/consequences

# 2. Inhalt füllen — KEIN `backlog decision edit` vorhanden!
#    Variante A: Markdown-Datei direkt editieren (Body unter ## Context etc.)
#    Variante B: `backlog browser` öffnen → Decision-Seite → bearbeiten
#                (server: updateDecisionFromContent)

# 3. Status pflegen (proposed→accepted/rejected/superseded):
#    ebenfalls nur per Datei-Edit oder Web-UI

Heißt konkret: Decisions sind im Tooling am dünnsten ausgestattet. Kein edit in der CLI, keine MCP-Tools. Für einen agentengetriebenen Flow ist das eine echte Reibung — der Agent muss die Datei selbst schreiben (was laut deiner Repo-Regel „nie Task-Files direkt editieren" nur für Tasks gilt; für Decisions ist Datei-Edit der vorgesehene Weg).

→ Wie kommen Documents rein?

backlog doc create "API Guidelines" -t guide -p guides/setup
backlog doc update doc-1 --content "..." --tags setup,runbook
backlog doc view doc-1   #   backlog doc list

Documents haben — anders als Decisions — volles CRUD über CLI und MCP und Web. Genau deshalb sind sie der bessere Kandidat für „evolvierendes Wissen" (siehe §6).

5. Vergleich: Backlog vs. dein aktueller knowledge/-Ansatz

Dein heutiges System (aus knowledge/README.md):

  • lektion-<thema>.md — Lessons-Learned aus Session-Analyse (evidenzlastig: Token-Tabellen, Fehlerzahlen, Code-Refs).
  • standardisiertes-vorgehen-fuer-<thema>.md — „How we do X"-Guides.
  • *.html — visuelle Analysen mit Live-Mermaid-Diagrammen.
  • Referenziert aus CLAUDE.md via einzeiligen „Knowledge Index" (on-demand geladen, bläht nicht jeden Turn).
  • „Decision-Changelog" = ad-hoc, kein dediziertes Format.

Der entscheidende Punkt: Backlog-decisions sind ADRs, kein Lessons-Learned-Format. Eine ADR sagt „wir haben X statt Y gewählt, weil Z, mit diesen Konsequenzen" — vorausschauend, punkt-in-zeit, append-only (superseded statt gelöscht). Deine lektion-*.md sagen „dieser Fehler trat N× auf, hier Muster + Fix" — rückblickend, evidenzbasiert, evolvierend. Das sind verschiedene Genres. Lessons-Learned in decision zu zwängen verbiegt beide.

6. Empfehlung: „Eine Quelle pro Repo" — ja, aber als Hybrid

„Eine Quelle für Task- und Knowledge-Management pro Repo via Backlog" ist technisch sauber machbar, weil Backlog-Docs/Decisions nur Markdown-Dateien im Git sind — grep-/qmd-indexierbar, kein Lock-in. Der einzige reale Kostenfaktor ist eine doppelte Taxonomie (knowledge/ vs. backlog/docs/), wenn man beides parallel führt. Daher: nach Genre zuordnen, nicht pauschal migrieren.

① ADRs → backlog decision

Echte „wir haben X entschieden"-Records. Gewinnt Nummerierung, Status-Lifecycle, Web-UI, gemeinsame Suche. empfohlen Ersetzt deinen ad-hoc „Decision-Changelog" sauber. Reibung: Body per File/Web füllen (kein CLI-edit).

② How-we-do-X-Guides → backlog doc (type: guide)

Deine standardisiertes-vorgehen-fuer-*.md passen 1:1. Volles CRUD über CLI+MCP, evolvierbar, im Browser sichtbar, durchsuchbar. empfohlen

③ Lessons-Learned (lektion-*.md) → knowledge/ behalten oder backlog doc

Passen nicht in decision (Genre-Mismatch). Zwei valide Wege: (a) als backlog doc (type: other/guide) führen, wenn du alles im Web-Browser und in backlog search willst — Docs sind editierbar/evolvierend, was Lessons brauchen. (b) In knowledge/ belassen, wenn dir das CLAUDE.md-„on-demand-Load"-Muster wichtiger ist als die Web-UI. deine Wahl

④ Interaktive HTML/Mermaid-Analysen → bleiben Files

Backlog rendert nur Markdown, kein arbiträres HTML mit Live-Mermaid. Diese Datei selbst (und session-lifecycle-hooks-analyse.html) gehören weiter in knowledge/ oder als statische Artefakte. Alternativ Mermaid als ```mermaid-Codeblock in ein doc — wenn die Web-UI Mermaid rendert (nicht garantiert, vorher prüfen). nicht migrierbar
