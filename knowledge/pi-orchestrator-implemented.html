cc-setup · pi-Orchestrator · Ist-Stand

Gated Agentic Workflow
// wie aktuell implementiert (CCS-035 / CCS-036)

Ein zweistufiger Loop: ein schwaches lokales Modell (pi/gemma) sequenziert mechanisch, delegiert jeden intelligenten Schritt an Claude-Worker, und der Mensch entscheidet an jedem kritischen Gate — per pi-Eingabebox ODER Slack. Deterministische Gates und Caps sichern den Rahmen.

Schwaches Modell orchestriert · starkes Modell arbeitet · Mensch gibt frei. Push nur auf Branch, nach gesehener Diff.
pi / gemma (Orchestrator) Claude-Worker (Agent SDK) Mensch (Slack / pi-Fenster) deterministisch (Script, kein LLM)
00

Wurde es so genutzt wie im Konzept-Dokument?

Das Dokument gated-agentic-workflow.html beschreibt das Konzept aus dem Talk (ADR/PRD/BDD/Cucumber, Git-Hook-Gates on commit/push/PR, ein Agent im Harness). Die cc-setup-Implementierung übernimmt den Geist, weicht in der Mechanik aber bewusst ab:

JA
Backlog.md als Management-Layer. Spec = Milestone, darunter Subtasks als Arbeitseinheiten, Fortschritt zweistufig (done/total je Milestone). 1:1 wie beschrieben.
JA
Gated & fail-fast. Mensch gibt Regeln/„Warum" vor, Agenten arbeiten im gesicherten Rahmen, ein deterministisches Gate + Reviewer erzwingen Qualität; rotes Gate → Task nicht Done.
TEILS
Der „Agent". Im Talk: ein Agent im Harness. Hier: zwei Stufen — ein schwaches lokales pi/gemma als mechanischer Sequencer + austauschbare Claude-Worker (planner/builder/reviewer/milestone-planner) via Agent SDK.
TEILS
Das Gate. Statt Git-Hooks on commit/push/PR + Cucumber: ein deterministisches run-gates.sh (just/npm/cargo/deno test, auto-detektiert) am DONE-Schritt + ein Reviewer-Worker. Keine BDD/Cucumber-Schicht.
NEU
Explizite Human-Gates. Über das Konzept hinaus: Spec-Freigabe (GATE₀), Cap-Override, Publish- und Weiter-Freigabe — jeweils via Slack ODER pi-Fenster. Mehr Human-in-the-loop als im Talk.
ANDERS
Merge. Kein automatischer PR-Merge. Der Orchestrator pusht nach Freigabe auf einen Feature-Branch pio/<id> (nie main, nie force); der Merge nach main bleibt manueller Mensch-Schritt.
FEHLT
ADR/PRD/BDD-Spec-Kette + Cucumber-„executable spec" + die UI/Test/ADR-Skills sind nicht Teil dieser Implementierung. Spec = Backlog-Task-Beschreibung + Planner-Output + Human-GATE₀.

⊟ Kurzfassung

Backlog ja, exakt wie beschrieben. pi: ja, aber als zweistufiger Orchestrator (nicht der Einzel-Agent des Talks). Das Gate ist run-gates.sh + Reviewer statt Hook/Cucumber, und es gibt deutlich mehr explizite Slack/TUI-Freigaben. Die Kernidee „Mensch setzt Rahmen, Agent arbeitet gated, fail-fast" ist erfüllt.

01

Drei Akteure — wer macht was

Die Aufgabentrennung ist der Kern: das billige lokale Modell trifft keine inhaltlichen Entscheidungen, das teure Modell sieht nur seinen eng umrissenen Auftrag, der Mensch gatet.

pi / gemma

Orchestrator — lokal (ollama, gemma4:12b), läuft als pi-Extension cc-orchestrator.ts. Mechanischer Sequencer, keine Code-Tools.

  • 6 Tools: backlog_next, backlog_set, dispatch_worker, run_gates, request_human, mark_done
  • folgt nur der Pipeline (STEP 1–8), improvisiert nicht
  • Tool-Lock via setActiveTools

Claude

Worker — via Claude Agent SDK (cc-dispatch.ts, query()), isoliert (settingSources:[]), Permissions über canUseTool-Whitelist.

  • planner — Plan + AC + Dateien + Risiken (read-only)
  • builder — implementiert, committet lokal, kein push
  • reviewer — APPROVE/REJECT + Findings, fährt Tests
  • milestone-planner — zerlegt neuen Meilenstein in Drafts

Mensch

Gatekeeper — antwortet an jedem Gate im pi-Fenster ODER via Slack (was zuerst kommt). Fallback bei Timeout: ntfy-Push + Datei-Flag-Resume.

  • Spec-Freigabe (GATE₀), Cap-Override
  • Publish-Freigabe (sieht den Diff), Weiter-Freigabe
  • Merge nach main (immer manuell)
02

Die Pipeline — Schritte & Bedingungen

Ein Task durchläuft STEP 1–8. STEP 7/8 stecken im mark_done-Toolcall (robust gegen gemma-Hänger). Farbe = wer den Schritt fährt.

1
PICK

PICK pi

backlog_next → nächster To Do-Task, gescoped auf CC_ORCH_MILESTONE; [Meilenstein]-Parents werden übersprungen. Dann backlog_set In Progress.

Bedingung: kein offener Task → STOP („no actionable tasks").
2
SPEC

SPEC Claude planner

dispatch_worker(planner) → nummerierter Plan, geschärfte AC, zu ändernde Dateien, Risiken.

Bedingung: Planner gibt "OPEN QUESTION" aus → request_human(OPEN_QUESTION) → Antwort einarbeiten, dann GATE₀.
3
GATE₀

GATE₀ · Spec-Freigabe Mensch · PFLICHT

request_human(GATE0_SPEC) — der vollständige Planner-Plan wird mitgeschickt (nicht nur 2 Zeilen).

approve → STEP 4 · Freitext → einarbeiten, GATE₀ erneut · cancel → STOP · PAUSED (Slack-Timeout) → STOP, Resume via Datei-Flag
4
DEV

DEV + GATE₁ Claude builder run_gates

dispatch_worker(builder) implementiert + committet lokal → run_gates (deterministisch). Diff der geänderten Dateien wird angehängt.

PASS/SKIP (keine Gates konfiguriert) → STEP 5 · FAIL → builder-Fix, erneut run_gates (bis MAX_DEV_RETRIES=2), dann request_human(GATE1_RETRIES)
5
REVIEW

REVIEW Claude reviewer

dispatch_worker(reviewer) → Diff-Review gegen AC, fährt Tests. Separate Session vom builder (dev≠review).

APPROVE → STEP 6 · REJECT → builder-Fix → run_gates → erneut Review (bis MAX_REVIEW_RETRIES=1), dann request_human(REVIEW_RETRIES)
6
DONE

DONE pi · mark_done re-check

mark_done prüft Gates erneut code-seitig, hakt AC ab, setzt Final-Summary, Status → Done.

Bedingung: Kill-Switch/Cap aktiv ODER Gate rot → Done NICHT gesetzt, bleibt In Progress.
7
PUBLISH

PUBLISH · Push-Freigabe Mensch pi push

In mark_done: Diff via Slack/TUI → request_human(PUBLISH).

approve → publishTask: dirty? gezielt nachcommitten (kein -A); git branch -f pio/<id> + push -u origin — nie main, nie force · sonst → kein Push (Task bleibt Done, lokal committet)
8
NEXT

NEXT · Weiter-Freigabe Mensch

In mark_done: nächsten To Do peeken (+ Beschreibung) → request_human(NEXT_TASK).

approve → CONTINUE_NEXT → zurück zu STEP 1 (↻ Task-für-Task) · nein / kein nächster Task → PIPELINE_DONE → STOP

↻ Der Loop

PICK → SPEC → GATE₀ → DEV+GATE₁ → REVIEW → DONE → PUBLISH → NEXT — und bei „weiter" zurück zu PICK. So wird ein ganzer Milestone Task für Task abgearbeitet, mit menschlicher Freigabe an Spec, Push und Übergang.

03

Gates & Bedingungen

Zwei deterministische Qualitäts-Tore (kein LLM) plus der Reviewer. „Done" ist code-erzwungen, nicht modell-behauptet.

GateWerWie / Bedingung
GATE₁ · run_gatesScriptrun-gates.sh auto-detektiert: justfile→just test · package.json→npm test · Cargo.toml→cargo test · deno.json→deno task test · oder .pi/gates. Keine Gates konfiguriert → SKIP = PASS (lautes Log; ADR 008), damit Doku/Sandbox-Repos nicht fail-closed hängen.
REVIEWreviewerEigener Claude-Worker prüft Diff gegen AC, Bugs, Security, Patterns, fährt Tests. VERDICT: APPROVE|REJECT + nummerierte Findings. Separate Session vom builder.
mark_donepiVor Done: Gates erneut grün (sonst BLOCKED), AC abgehakt, Final-Summary gesetzt. Bei rotem Gate/Cap/Kill-Switch bleibt der Task In Progress.
04

Human-Gates — via Slack oder pi-Fenster

Jede request_human-Frage läuft als Race: Slack-Poll (Kanal C0B8R3ERUNR) und lokale pi-Eingabebox (ctx.ui.input) gleichzeitig — die erste Antwort gewinnt, der Verlierer wird abgebrochen. Antwort wird klassifiziert: ja/ok/approved · nein/cancel/abort · Freitext.

PhaseWannAntwort bewirkt
OPEN_QUESTIONPlanner ist sich unsicher (STEP 2)Freitext/approve → in Plan einarbeiten → GATE₀ · abort → STOP
GATE0_SPECSTEP 3 · immer (Pflicht)approve → DEV · Freitext → re-gate · cancel → STOP. Voller Plan ist angehängt.
GATE1_RETRIESGates nach 2 Builder-Versuchen rotOverride → ein weiterer Versuch · cancel/PAUSE → STOP
REVIEW_RETRIESReviewer nach 1 Fix-Zyklus weiter REJECTOverride → ein weiterer Builder+Review-Zyklus · cancel → STOP
CAP_EXCEEDEDRetry-/Kosten-/Zeit-Cap überschrittenOverride → genau ein Schritt weiter (begrenzt, s. §05)
CAP_OVERRIDE_LIMITnach 2 Overrides ohne Fortschrittkein weiterer Override → harte Pause (pausePipeline)
PUBLISHSTEP 7 · nach Done, Diff angehängtapprove → Push auf pio/<id> · sonst kein Push
NEXT_TASKSTEP 8 · nächster Task + Beschreibungapprove → Loop zu STEP 1 · nein → Pipeline-Ende

⏸ Fallback, wenn niemand antwortet

Slack-Timeout (900 s) und keine lokale Eingabe → pausePipeline: schreibt .pi/orchestrator-state.json, sendet ntfy-Push, gibt HUMAN_REQUIRED/PAUSED zurück. Fortsetzen mit touch .pi/orchestrator-resume (oder …-resume-cancel).

05

Caps, Kill-Switch & Isolation

Code-seitig erzwungen in cc-orchestrator.ts — nicht modellabhängig. Pro Task in backlog_next zurückgesetzt.

Retries & Budget

  • MAX_DEV_RETRIES = 2
  • MAX_REVIEW_RETRIES = 1
  • MAX_COST_USD_PER_TASK = 5
  • MAX_WALL_CLOCK = 900 s
  • MAX_CAP_OVERRIDES = 2 → dann harte Pause

Stop-Mechanik

  • Kill-Switch: touch .pi/orchestrator.kill
  • Single-flight-Lock: .pi/orchestrator.lock
  • Cap/Kill aktiv → mark_done BLOCKED

Worker-Isolation

  • settingSources: [] (keine ~/.claude-Hooks)
  • permissionMode: 'default' + canUseTool
  • builder/reviewer dürfen kein git push
  • Egress: redactor in der Hauptsession, nicht im Worker
06

Start: pio / just pi — Meilenstein-Auswahl

Der Launcher pi-launch.sh ermittelt offene Meilensteine (sprint_bridge survey) und baut den pi-Prompt programmatisch. Neuer Meilenstein wird vor dem Build per Claude-Worker in Aufgaben ausgearbeitet.

AuswahlWas passiert
Bestehender MeilensteinCC_ORCH_MILESTONE wird gesetzt → Pipeline gescoped auf diesen Meilenstein → exec pi.
Tasks direktNur bei 0 offenen Meilensteinen: globaler PICK ohne Scope (statt Neu-Zwang). Auch robust, wenn sprint_bridge leer/fehlerhaft ist.
Neuer Meilensteinmilestone-planner-Worker (Claude, sonnet) zerlegt das Ziel in 3–7 Draft-Tasks (backlog task create --draft -m) → Human-Gate: Drafts ansehen → promote (To Do) oder archive → CC_ORCH_MILESTONE → exec pi.
