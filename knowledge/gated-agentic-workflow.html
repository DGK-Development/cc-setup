Gated Agentic Workflow
// PRD-019: Scheduled eval runs

Der Mensch gibt Regeln und das „Warum“ in Dokumenten vor. KI-Agenten arbeiten autonom innerhalb eines durch Git-Hooks und CI gesicherten Rahmens — und werden bei Fehlern durch automatisiertes Feedback zur Selbstkorrektur gezwungen.

If you can't measure it, you can't enforce it.
00

Der Kreislauf

Erfassung → Durchsetzung → iterative Korrektur. Schlägt im Gate ein Check fehl, geht das Feedback fail fast zurück an den Agent. Erst wenn alle Stufen grün sind, wird gemerged.

01

Erfassung von Entscheidungen

Bevor Code geschrieben wird, werden die Rahmenbedingungen maschinen- und menschenlesbar dokumentiert. So weiß sowohl der Mensch als auch der Agent, warum etwas gilt — nicht nur, dass es gilt.

ADRArchitecture Decision Records

Hält fest, warum eine Architektur-Entscheidung getroffen wurde — etwa die Trennung von Schichten zur Vermeidung von N+1-Abfragen — und wie die Regel technisch erzwungen wird.

PRDProduct Requirements Document

Leichtgewichtig. Beschreibt den Zweck eines Features, das Ziel und die User Journey.

BDDBehavior-Driven Development

Eine Zwischenschicht in menschlicher Sprache (ausführbarer Markdown via Cucumber), die beschreibt, wie sich das Produkt verhält. Für Menschen leichter zu prüfen als reiner KI-Testcode.

DSDesign-Systeme

UI-Regeln (Farben, Button-Größen) als Code-Snippets bereitgestellt. Agenten bauen konsistente Oberflächen, statt bei jedem Element neu zu improvisieren.

  prd-019.md

03

Feedback und Iteration

Das Herzstück des gated Mechanismus. Statt Aufsicht ersetzt das Gate Review durch harte, sofortige Rückmeldung — und kanalisiert das Verhalten des Agents.

Ablehnung bei Fehlern

Erfüllt der Code die Regeln nicht, wird der Commit abgelehnt — am Commit-Zeitpunkt, nicht erst im Review.

Kontextbezogenes Feedback

Der Agent erhält eine Fehlermeldung, die ihn direkt auf das entsprechende Dokument (z. B. das ADR) zurückverweist.

Korrektur in neuer Iteration

Der Agent liest das Dokument erneut, versteht den Grund für die Regel und korrigiert seinen Code. „Agent gets early feedback and works through it to reach the goal.“

04

Spezialisierte Skills für den Fokus

Verschiedene Aufgaben haben unterschiedliche Schwerpunkte. Skills passen den Loop an die jeweilige Tätigkeit an. (Anklicken zum Aufklappen.)

UI-Skill Schnelle Iteration im Browser ›
Überspringt bestimmte Backend-Prüfungen, um schnelle Iterationen direkt im Browser zu ermöglichen. Greift auf die Snippets des Design-Systems zurück, damit Oberflächen konsistent bleiben.
Test-Skill Gezielte Testauswahl statt voller Suite ›
Identifiziert anhand von Code-Abdeckung und Dateiänderungen genau die Tests, die ausgeführt werden müssen — statt die gesamte Suite zu starten. Spart Zeit und schließt den Loop schneller.
ADR-Skill Betroffene Architektur-Dokumente finden ›
Hilft dem Agenten, gezielt die Architektur-Dokumente zu finden, die von seinen aktuellen Änderungen betroffen sind — damit Feedback und Korrektur am richtigen ADR ansetzen.
05

Backlog.md als Management-Layer

Die Spec sagt was & warum, das Gate sagt ob die Regeln halten — aber nichts orchestriert die Arbeitseinheiten dazwischen. Backlog.md ist genau diese Schicht — und seine Hierarchie passt 1:1: je Spec ein Milestone, darunter Subtasks, die abgearbeitet und in PRs gebündelt werden.

npm i -g backlog.md

Verwaltet jedes Issue als reine .md-Datei im Repo. 100 % offline, spec-driven, mit „Definition of Done"-Checklisten pro Task. Spricht Agenten via MCP oder CLI an.

↳ Jede Spec wird ein Milestone. Subtasks darunter sind die Arbeitseinheiten; ein PR bündelt eine Subtask-Gruppe und läuft durchs Gate.

Andockpunkte je Phase

Phase 01Erfassung
backlog milestone · task --parent · --ac

PRD-019 wird ein Milestone. Der Agent zerlegt die User Journey in Subtasks mit Acceptance Criteria, die den BDD-Szenarien entsprechen. Der decisions-Bereich wird zum CLI-durchsuchbaren ADR-Store.

Phase 02The Loop
Subtasks → 1 PR → Gate

Der Agent arbeitet die Subtasks eines Milestones ab und bündelt sie in einem PR. Die Gate-Schleife sitzt auf PR-Ebene — kohärenter Change, aber klein genug für präzises fail fast. Der Plan-Schritt vor dem Coden ist dein Review-Checkpoint.

Phase 03Feedback
definition_of_done

Die DoD-Checkliste (Tests pass · No regressions) ist die menschlich lesbare Brücke: Gate rot → DoD nicht erfüllt → Task bleibt offen. Bei schlechtem Output: Plan löschen, Task schärfen, frische Session.

Phase 04Skills
backlog search --modified-file

Tasks speichern die geänderten Dateien. Der Test-Skill findet so genau die Tasks/Tests, die ein Change berührt; der ADR-Skill sucht über den decisions-Bereich.

  setup & loop
# 1 · Init im Repo + MCP-Anbindung (Claude Code / Codex / Gemini) $ backlog init "Scheduled Evals" $ claude mcp add backlog --scope user -- backlog mcp start # 2 · Spec als Milestone anlegen, Journey in Subtasks zerlegen (Agent, via MCP) # PRD-019 → Milestone "scheduled-evals" # └─ Subtasks: connect-model · set-schedule · alert-on-regression # 3 · Subtasks abarbeiten, in einem PR bündeln → Gate $ backlog task edit BACK-12.1 --ac "connect model" -s "Done" $ backlog task edit BACK-12.2 --ac "set schedule" -s "Done" # Agent: Plan → implementieren → ein PR für die Subtask-Gruppe # 4 · Fortschritt auf zwei Ebenen sichtbar machen $ backlog board # Subtasks pro Milestone, live $ backlog search --modified-file src/scheduler.ts --plain

⊹ Der entscheidende Gewinn

Der Milestone (= die Spec) bleibt die stabile Klammer, die Subtasks sind die abarbeitbaren Einheiten. Der Fortschritt wird damit auf zwei Ebenen lesbar: pro Milestone, wie viel der Spec erfüllt ist (X von Y Subtasks done), und pro PR, ob der Change durchs Gate ist. Der Milestone gilt erst als erledigt, wenn alle seine Subtasks Gate + DoD bestanden haben — das ist dein „PRD erfüllt"-Signal.
