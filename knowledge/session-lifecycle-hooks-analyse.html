cc-setup — Session-Lifecycle, Wissens-Architektur & Zielbild

Wie eine Session abläuft (SessionStart → UserPromptSubmit → Tool-Loop → Stop), woher Tasks kommen (Backlog vs. Vault-tn), welche Skills automatisch triggern, und wie Wissen pro Repo vereinheitlicht wird. Mit konkretem Zielbild für globale & repo-lokale CLAUDE.md.  Stand: 2026-06-05 (aktualisiert auf die Flat-Migration; ersetzt die Plugin-Doppel-Analyse vom 2026-06-04).

Leitprinzip: Einmal initial laden, dann gezielt aus Rule-Files sammeln — statt jedes Mal neu zu suchen.
Always-on (jeder Start) bleibt minimal: Backlog-Stand + schlanke Verhaltens-Leitplanken. Detail-Wissen (Konventionen, Entscheidungen, Lessons, Specs) liegt in Rule-Files (Skills, knowledge/, backlog/{decisions,docs}, Vault-Specs) und wird on-demand über /context-load + qmd gezogen, wenn die Aufgabe es verlangt.
Migrations-/Deployment-Hinweis: Dieses Dokument zeigt den Flat-Ziel-Zustand (Quelle ist nach CCS-005/CCS-012 flach im Repo-Root, Submodul aufgelöst, Hooks aus einer Quelle in ~/.claude/settings.json). Die alte Analyse beschrieb doppelt registrierte Hooks (settings.json und Plugin) → inject-project-context + /context-load feuerten 2×. In der flachen Quelle ist das aufgelöst. Solange die Flat-Commits aber noch nicht deployed sind (kein just deploy + Restart), kann das laufende ~/.claude real noch den alten Doppel-Zustand fahren. Verifikation nach Deploy: genau ein SessionStart-Block, ein context-load-Trigger, jeder Skill einmal gelistet.

4. Skills — automatisch vs. manuell

cc-setup deployt 12 tier-1-Skills (knowledge/skills.csv, quelle=projekt). Nur zwei Pfade triggern automatisch:

Trigger-ArtSkill(s)Wann
AUTO/context-load1. Prompt (UserPromptSubmit), genau 1×
AUTO (extern)using-superpowersSessionStart, voll injiziert (superpowers-Plugin)
manuell / modell-entschiedenaudit, review, qmd, recall, knowledge, daily-review, check-links, opensrc, local-ci, session-stop, cc-setupSlash-Befehl oder vom Modell per Description-Match gewählt

Optimale Skills je Phase: Start → context-load (auto). Verstehen/Suchen → qmd, recall, opensrc. Bauen → tdd, diagnose, karpathy-guidelines. Doku/Wissen → knowledge, doc-coauthoring. Pre-Push → local-ci. Ende → session-stop (Stop-Hook-gestützt). Audit → audit (LOGS + CONFIG). Flat-Ziel: jeder Skill erscheint einmal in der Liste (kein /cc-setup:*-Doppel mehr).

6. Globale CLAUDE.md — was bleibt, was wird Pfad-Referenz

Regel: Always-loaded (jeder Turn) nur Verhaltens-Leitplanken + Pointer. Detail wandert in Rule-Files (Skills, Specs, knowledge/) und kommt on-demand. Es gibt zwei Schichten in ~/.claude/CLAUDE.md:

  • User-global (handgepflegt): Kern-Prinzipien, Setup, TaskNotes, Redactor.
  • cc-setup-managed-Block (<!-- BEGIN/END cc-setup -->, aus CONTRACT.md): Runtime-Vertrag + SPOC-Routing — schon kompakt, bleibt.

HOCH S0 — Flat deployen & Doppel verifizieren

Nach Review: just deploy + Neustart. Verifizierbar: ein SessionStart-Block, ein context-load-Trigger, jeder Skill 1× gelistet. Erst danach stimmt das laufende System mit diesem Dokument überein.

MITTEL S1 — Global CLAUDE.md: Karpathy auf Headlines kürzen

~/.claude/CLAUDE.md · ## Kern-Prinzipien - <~35 Zeilen Detail-Bullets je Prinzip> + 5 Headlines (Think Before Coding · YAGNI · Surgical · Verifiable · Tradeoffs) + Detail: Skill `karpathy-guidelines`

MITTEL S2 — Global CLAUDE.md: tn-Arbeitsweise auf Kern + Pfad

~/.claude/CLAUDE.md · ## TaskNotes (tn) - Workflow-Block + Status-Pipeline + Command-Liste (~35 Zeilen) + tn global · CWD-Auto-Detection · nur User setzt done + Spec: Atlas/Wiki/.../tasknote-frontmatter-syntax-v1.md · Vault CLAUDE.md

MITTEL S3 — Global CLAUDE.md: Redactor auf 3 Zeilen + Lektion

~/.claude/CLAUDE.md · ## Redactor (Strict Mode) - Fallstricke/Config/Knowledge-Lookup (~45 Zeilen) + strict aktiv · `r wrap -- <cmd>` · strukturierte Files via `--type` + Fallstricke: knowledge/lektion-redactor-strict-mode-*.md

DONE/PILOT S4 — Wissens-Hybrid ausrollen

decision-001 + doc-001 sind in cc-setup pilotiert. erledigt hier; nächster Schritt: dasselbe Genre-Schema in einem weiteren Repo anwenden (1 ADR + 1 Guide als Pilot), bevor Bestands-knowledge/-Inhalte migriert werden.

NIEDRIG S5 — knowledge-Index als Deploy-Standard

knowledge/README.md-Konvention (1-Zeilen-Index) existiert. Vorschlag: in organize-project/Deploy als Template verankern, damit jedes neue Repo den minimal-CLAUDE.md-+-Index-Pattern erbt.

Netto-Effekt S1–S3: die global jeden Turn geladene CLAUDE.md schrumpft um ~115 Zeilen Detail auf Leitplanken + Pointer — bei gleicher Wirkung, weil das Detail on-demand aus den Rule-Files kommt. Genau „initial minimal laden, dann gezielt sammeln".
