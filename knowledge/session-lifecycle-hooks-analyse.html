cc-setup — Session-Lifecycle, Hooks & Skills

Analyse der Hook-Verdrahtung über den Session-Lebenszyklus (SessionStart → UserPromptSubmit → Tool-Loop → Stop), welche Skills automatisch getriggert werden, und Optimierungen unter der Leitlinie weniger ist besser.  Stand: 2026-06-04.

TL;DR — der größte Hebel: Hooks sind doppelt registriert (einmal in ~/.claude/settings.json, einmal im cc-setup-Plugin hooks/hooks.json). Dadurch feuern inject-project-context (SessionStart) und der /context-load-Trigger (UserPromptSubmit) zweimal pro Session — doppelte Token-Injektion, doppelte Latenz. Empirisch belegt: zwei „SESSION START HOOK"-Zeilen und zwei <context-load-trigger>-Blöcke beim Start. Eine Registrierungsquelle reicht.

7. Optimierungen — weniger ist besser

HOCH R1 — Hooks aus EINER Quelle registrieren

settings.json und Plugin registrieren denselben Hook-Satz → alles feuert doppelt. Eine Quelle wählen. Da Skills künftig flat laufen sollen: Hooks in ~/.claude/settings.json behalten, die Hook-Registrierung im Plugin (hooks/hooks.json) entfernen — oder umgekehrt.
Verifizierbar: nach dem Fix exakt ein SessionStart-Block und ein context-load-Trigger.

HOCH R2 — /context-load nur einmal

Direkte Folge von R1: spart die größte wiederkehrende Token-Injektion pro Session-Start (zwei nahezu identische Trigger-Blöcke → einer).

MITTEL R3 — SessionStart-Output verschlanken

inject-project-context.sh druckt jeden Start einen Project Tree (bis 30 Verzeichnisse + 15 Dateien). Bei großen Repos viel Rauschen für wenig Nutzen. Vorschlag: 1 Level / max. 10 Einträge, oder ganz weglassen (Claude kann bei Bedarf selbst listen).

MITTEL R4 — superpowers-Injektion prüfen

Das volle using-superpowers-Skill wird bei jedem SessionStart injiziert (mehrere KB). Wenn es nicht aktiv die Arbeitsweise steuert: deaktivieren oder auf einen Einzeiler-Pointer kürzen → Token-Ersparnis jeden Start.

NIEDRIG R5 — Skill-Namespace-Dopplung auflösen

Wenn das Plugin nur noch Hooks liefern soll: skills/ aus dem Plugin entfernen (nur hooks/ behalten). Dann erscheint jeder Skill nur einmal (flat) statt zweimal → kürzere Skill-Liste, weniger Auswahl-Rauschen.

NIEDRIG R6 — session-analyser SKILL.md Step 2 fixen

redactor wrap -- cat /tmp/session-agg.json wird unter Strict-Mode geblockt. Auf „JSON-Stream durch Plain-Text-Formatter pipen" umstellen (verifiziert funktionierend).

NIEDRIG R7 — Stop-Chain bewusst halten

sync-sessions-to-vault.sh läuft nur beim 2. Stop (nach PKM-Block). Funktioniert, aber unintuitiv — dokumentieren oder Reihenfolge prüfen.

Netto-Effekt von R1+R2+R5: ~halbe SessionStart/Prompt-Hook-Last, ein context-load statt zwei, halbierte Skill-Liste — bei identischer Funktion. Genau „weniger ist besser".
